# 0039. 공개 Resume은 요약 4열 그리드로 제공한다

Date: 2026-07-20
Status: Accepted

## 배경

기존 Resume은 프로젝트의 문제, 결정, 구현, 검증, 결과와 보조 이력을 한 페이지에
모두 노출했다. 내용 자체는 상세 이력서에 보존할 가치가 있지만, 채용담당자와
개발자가 처음 방문했을 때 역할과 적합성을 빠르게 파악하기에는 밀도가 높았다.

참조한 zero.log Resume은 콘텐츠 열에서 신원, 역할, 연락처를 먼저 보여준 뒤 회사와
핵심 결과를 일정한 열에 배치한다. Ark도 상세 이력서의 대체물이 아니라, 대화와
상세 검토로 이어지기 위한 공개 요약본이 필요하다.

## 결정

- 공개 `/resume`은 shell의 콘텐츠 열에 4열 요약 그리드를 사용한다. 모바일에서는
  2열로 전환한다.
- 첫 행은 ID, Role, Focus, Contact만 표시한다. 생년월일, 전화번호, 블로그 URL은
  공개 요약본에서 제외한다.
- 경력은 모노리스와 엑심베이 두 회사만 표시한다. 각 회사에는 역할, 기간, 세 개의
  결과 키워드만 둔다.
- 핵심 결과는 검증 가능한 변화나 범위를 짧게 적는다. 긴 프로젝트 단계와 서술형
  자기소개는 이 화면에서 렌더링하지 않는다.
- 역량은 Server systems, Data platform, Operations automation 세 묶음으로만
  정리한다.
- 이력 원본 데이터는 `resume/model/resume-data.ts`에 유지한다. 공개 요약본은
  별도의 summary data를 사용한다.

## 결과

- 첫 화면에서 백엔드, 데이터 플랫폼, 운영 자동화의 적합성을 빠르게 판단할 수
  있다.
- 상세 이력의 내용은 삭제되지 않으며, 별도의 상세 이력서나 후속 대화에 사용할 수
  있다.
- Resume은 홈과 Archive가 사용하는 동일한 2-6-4 shell 안에서 더 짧은 문서로
  읽힌다.

## 검토한 대안

- 기존 상세 Resume을 유지한다: 이력의 증거는 풍부하지만 첫 화면의 판단 비용이
  높다.
- 상세 정보를 아코디언으로 접는다: 정보량은 유지하지만 사용자가 무엇을 열어야
  하는지 먼저 판단해야 한다.
- 공개 Resume을 요약 그리드로 바꾼다: 정보량을 의도적으로 줄이지만 역할, 도메인,
  성과를 한 번에 비교할 수 있다.

## 검증

- ResumePage 컴포넌트 테스트로 역할, 연락처, 두 회사, 핵심 결과를 검증한다.
- 컴포넌트 테스트로 기존 장문 섹션이 렌더링되지 않는 것을 검증한다.
- Playwright로 데스크톱 4열과 모바일 2열 레이아웃을 확인한다.
