# 블로그 품질 가이드

## 브랜드 문장

새로운 비즈니스 확장으로 복잡해지는 내부 시스템을 정리하고, 레거시를 구조적으로 개선하는 플랫폼 엔지니어

## 기본 원칙

- 기술 글은 감상보다 판단 기준이 먼저 보여야 한다.
- 구현 자랑보다 문제 정의와 대안 비교가 먼저 나와야 한다.
- 글 하나가 읽히고 끝나면 안 된다. 같은 문제를 다시 만났을 때 재사용 가능한 기준이 남아야 한다.
- 공개 글은 포트폴리오다. 초안이나 메모 수준의 글은 `private`로 둔다.

## 독자 타깃 프레임워크

글을 시작하기 전에 6W 1H를 채운다. 모든 항목을 본문에 직접 드러낼 필요는
없지만, 제목과 도입, 소제목, 결론은 이 답변을 기준으로 설계한다.

- `Who`: 이 글을 가장 필요로 하는 독자는 누구인가
- `Whom`: 이 문제의 영향을 받거나 결과를 소비하는 사람은 누구인가
- `What`: 독자가 해결하려는 문제, 궁금증, 판단은 무엇인가
- `When`: 독자가 이 글을 찾게 되는 시점은 언제인가
- `Where`: 독자가 처한 업무, 생활, 시스템, 조직 맥락은 어디인가
- `Why`: 지금 이 문제를 해결하지 않으면 어떤 비용이나 불편이 생기는가
- `How`: 글을 읽고 나서 독자가 어떤 방식으로 적용하거나 판단할 수 있는가

적어도 `Who`, `What`, `Why`, `How`가 분명하지 않으면 공개 글로 올리지
않는다. 이 네 가지가 약하면 좋은 문장이어도 정확한 독자에게 꽂히기 어렵다.

도입 3문단은 아래 흐름을 우선한다.

1. `Who/When/Where`: 어떤 독자가 어떤 상황에 놓여 있는가
2. `What/Why`: 그 상황에서 무엇이 문제이고 어떤 비용이 생기는가
3. `How`: 이 글이 어떤 기준이나 방법을 남기는가

## 기술 글 톤

- 기본 문체는 단정형 `하다체`로 쓴다.
- 도입과 마무리만 제한적으로 부드럽게 쓸 수 있다.
- "배웠다", "느꼈다"는 문장만으로 끝내지 않는다. 무엇이 바뀌었는지까지 적는다.
- 추상 표현 대신 수치, 조건, 실패 사례를 우선한다.

## 글 형식 분류

`category`는 주제 영역이고, `contentType`은 글의 형식이다. `Tech/Life`
category는 유지하고, 글이 독자에게 어떤 기대를 주는지는 `contentType`으로
구분한다.

- `essay`: 관점, 주장, 해석, 원칙, 설명형 가이드가 중심인 글.
- `retrospective`: 사건, 프로젝트, 장애, 이직, 면접처럼 실제 경험을 되돌아보고
  다음 기준을 남기는 글.
- `review`: 책, 여행, 제품, 장소, 외부 콘텐츠처럼 평가 대상이 분명한 글.

Life 글은 아래 기준으로 본다.

- `essay`: 관점이 선명하고, 감정이나 사유가 구체 장면과 연결되는가.
- `retrospective`: 사건, 전환점, 배운 기준, 다음 행동이 드러나는가.
- `review`: 평가 기준, 대상의 장단점, 추천할 독자와 추천하지 않을 독자가
  분명한가.

## 공개용 엔지니어링 글 템플릿

### 1. 문제와 비즈니스 맥락

- 왜 이 문제가 운영 또는 비즈니스 비용으로 이어졌는가
- 누가 이 결과를 소비하는가

### 2. 시스템 요구사항과 제약

- 성능, 정확성, 복구, 운영 조건
- 리소스 한계나 조직 제약

### 3. 대안 비교와 선택 근거

- 실제로 검토한 대안 2개 이상
- 버린 대안과 이유

### 4. 실제 아키텍처와 구현

- 데이터 흐름
- 핵심 상태 모델
- 중요한 설정값과 선택 이유

### 5. 실패와 수정

- 장애, 병목, 잘못된 가정
- 어떻게 관측했고 어떻게 수정했는가

### 6. 검증과 결과

- 단위 테스트
- 통합 테스트
- 운영 확인 방식
- 수치 결과

### 7. 재사용 가능한 판단 기준

- 다음에도 그대로 가져갈 기준
- 아직 남은 한계

## 공개 정책

- `visibility` 기본값은 `private`다. 공개는 리뷰를 마친 뒤 명시적으로 승격한다.
- `contentType`은 `essay`, `retrospective`, `review` 중 하나로 둔다.
- 시리즈는 기본적으로 공개 자산이 아니라 학습 자산으로 보고, 공개 필요성이 생기기 전까지 `private`로 둔다.
- 엔지니어링 글은 `philosophy`, `design`, `implementation` 평균이
  `blog/services/policy.ts`의 공개 하한을 넘어야 한다.
- `featured`는 아래 조건을 모두 만족할 때만 허용한다.
  - 엔지니어링 글
  - 시리즈 아님
  - `blog/services/policy.ts`의 `brandFit` 하한 충족
- `featured`는 점수만으로 자동 결정하지 않는다. 현재 단계에서는 수동 큐레이션 목록으로 관리한다.
- 현재 홈은 `featured` 글을 별도 노출하지 않고 전체 공개 글을 단일 피드로 보여준다.

## 글 점수

점수는 작성과 리뷰를 위한 내부 운영 도구다. 공개 UI에 바로 노출하지 않는다.
모든 점수는 `1~5`, `0.5` 단위로 기록하고, 판단할 근거가 부족하면 `null`로 둔다.

- `philosophy`: 재사용 가능한 판단 기준이나 관점이 남는가.
- `design`: 글의 구조와 흐름이 독자의 기대를 따라가는가.
- `implementation`: 실제 장면, 구현, 실패, 수치, 예시가 충분한가.
- `brandFit`: Ark의 공개 정체성과 작성자 포지셔닝에 맞는가.
- `clarity`: 주장, 용어, 결론, 독자 약속이 명확한가.
- `structure`: 문단과 섹션이 자연스럽게 누적되는가.
- `evidence`: 경험, 데이터, 조건, 비교 근거가 충분한가.
- `usefulness`: 독자가 가져갈 판단, 행동, 기준이 있는가.
- `originality`: 흔한 요약을 넘어 작성자 고유의 관점이 있는가.
- `polish`: 문장, 오탈자, 중복, 톤이 정리되어 있는가.

## 시스템 점수

글 점수와 별도로 블로그 시스템은 다음 축으로 본다. 이 점수는 post meta가 아니라
별도 리포트나 개선 계획에서 관리한다.

- UI: 화면 구성과 컴포넌트 완성도
- UX: 탐색, 필터, 읽기 흐름, 반복 사용성
- design: 브랜드 정체성, 레이아웃, 시각적 일관성
- discoverability: 검색, SEO, RSS, OG, 추천 흐름
- readability: 모바일 읽기, 목차, 문단 길이, 코드/이미지 배치
- performance: build 안정성, 렌더링 속도, asset 최적화
- accessibility: 대비, 키보드, 모션, semantic 구조
- maintainability: 글 추가, meta 관리, 테스트, 문서화
- trust/safety: private/public 정책, NDA/보안, 근거와 업데이트 이력

## 리뷰 체크리스트

- 6W 1H 중 `Who`, `What`, `Why`, `How`가 분명한가
- `contentType`이 독자 기대와 맞는가
- 첫 세 문단 안에 문제와 비용이 드러나는가
- 이 글을 가장 필요로 하는 독자가 제목과 도입에서 바로 보이는가
- 대안 비교가 실제로 존재하는가
- 설정값과 구조가 왜 그렇게 됐는지 설명하는가
- 장애 또는 실패 장면이 포함되어 있는가
- 결과가 수치 또는 운영 변화로 확인되는가
- 읽고 나면 한 문장으로 재사용 가능한 기준이 남는가
