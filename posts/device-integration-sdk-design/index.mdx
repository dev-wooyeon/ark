# 현장 장비 연동 SDK 설계: 표준화의 경계 정하기

현장 장비가 늘어나면 통신 라이브러리, 연결 설정, 재연결 처리, 메시지 형식도
함께 늘어난다. 처음에는 각 팀이 자기 장비를 빠르게 연결하는 편이 합리적으로 보인다.
하지만 장비 종류와 개발 언어가 늘어나면, 같은 통신 문제를 여러 팀이 반복해서 풀게 된다.

이 글은 메시지 브로커 기반 현장 장비 시스템을 SDK로 표준화할 때의 설계 판단을
정리한다. 여러 언어로 디바이스 에이전트를 개발하는 엔지니어가 SDK의 책임 범위,
메시지 계약, 명령 상태, 검증 환경을 결정하는 데 목적이 있다.

이 글에서 **디바이스 에이전트**는 현장 장비에서 SDK를 호출해 서버와 통신하는
애플리케이션을 뜻한다. 물리 장비와 그 위에서 동작하는 소프트웨어를 구분하기 위해
이 용어를 사용한다.

이 글에서 다룰 질문은 다섯 가지다.

1. 무엇을 공통 메시지 계약으로 고정할 것인가?
2. 명령 수신 확인과 실행 결과를 왜 구분해야 하는가?
3. 지연된 응답을 원래 명령과 어떻게 연결할 것인가?
4. 전송 구현과 다국어 SDK를 어떤 경계로 나눌 것인가?
5. 실제 장비가 없어도 어떻게 계약을 검증할 것인가?

> 공개 범위 안내: 이 글은 특정 운영 환경의 구현을 재현하지 않는다.
> 식별 가능한 운영 정보와 보안 설정은 제외했고, 예제 코드는 설명을 위한 의사 코드다.

## 문제: 디바이스 에이전트마다 통신 코드를 다시 만드는 구조

디바이스 에이전트와 서버를 메시지 브로커로 연결하는 시스템에서는 보통 이런 일이
일어난다.

- 디바이스 에이전트를 개발하는 팀마다 다른 클라이언트 라이브러리를 선택한다.
- 팀마다 연결 복구, 구독 복원, 오류 처리를 다시 구현한다.
- 메시지 형식과 명령 결과를 해석하는 방법이 조금씩 달라진다.
- 새 디바이스 에이전트를 추가할 때마다 플랫폼 팀에 물어봐야 한다.

겉으로는 라이브러리가 여러 개인 문제가 커 보인다. 더 근본적인 문제는
**디바이스 에이전트 개발자가 통신 인프라의 세부사항까지 알아야 한다는 것**이다.

SDK의 역할은 특정 메시지 라이브러리를 감싸는 데서 끝나면 안 된다. 애플리케이션은
장비의 업무와 메시지 계약을 알고, 연결 유지와 직렬화 같은 전송 세부사항은 SDK가 맡는다.

```text
디바이스 에이전트
  -> SDK의 메시지 계약
    -> 전송 어댑터
      -> 메시지 브로커
```

이 경계가 생기면 새 팀은 전송 설정을 복제하는 대신 정해진 계약으로 메시지를 만들고
핸들러를 등록한다. 표준화의 목표는 코드 모양을 같게 만드는 것이 아니라
**새 팀의 질문 수를 줄이는 것**이다.

## 결정 1: 전달 봉투는 고정하고 업무 데이터는 연다

SDK가 모든 것을 타입으로 고정하면 새 디바이스 에이전트나 새 동작이 추가될 때마다
SDK도 바뀐다.
반대로 아무것도 정하지 않으면 SDK는 단순한 연결 도구가 된다. 따라서 메시지에서
변하지 않는 부분과 장비마다 달라지는 부분을 먼저 분리해야 한다.

| 구분        | 예시                                             | 소유자            | 바뀌는 빈도 |
| ----------- | ------------------------------------------------ | ----------------- | ----------- |
| 전달 봉투   | 메시지 ID, 발생 시각, 계약 버전, 메시지 종류     | SDK               | 낮음        |
| 장비 정체성 | 장비 식별자, 설치 위치처럼 연결 중 고정되는 정보 | 초기화 설정       | 낮음        |
| 업무 의미   | 동작 이름, 업무 맥락, 실제 데이터                | 디바이스 에이전트 | 높음        |

원칙은 하나다.

> 안정된 구조는 SDK가 만들고, 자주 달라지는 업무 데이터는 호출자가 채운다.

이 글에서는 `event`, `command`, `ack`, `result`처럼 메시지의 목적을 제한된 집합으로
본다. 반면 디바이스 에이전트가 발행할 사건과 수행할 동작, 페이로드(payload)는
계속 늘어난다.
목적은 SDK 메서드로 고정하고 업무 동작은 인자로 남겨야, 새 동작을 추가해도 SDK를
다시 배포하지 않는다.

```ts
// 설명을 위한 의사 코드
const receipt = client.publishEvent({
  action: 'CHECK_IN',
  payload: {
    /* 장비별 데이터 */
  },
});
```

이 설계는 페이로드를 검증하지 말자는 뜻이 아니다. 업무 스키마는 해당 업무를 소유한
팀이 검증한다. SDK는 모든 팀이 함께 지켜야 하는 전달 봉투와 메시지 생명주기를 보장한다.

## 결정 2: 수신 확인과 실행 결과를 분리한다

장비 제어에서 가장 위험한 오해는 “응답을 받았다”와 “명령이 수행됐다”를 같은 상태로
보는 것이다.

```text
명령 전송
  -> 수신 확인(ack): 디바이스 에이전트가 명령을 받았는가?
    -> 실행 결과(result): 제어 동작이 실제로 수행됐는가?
```

수신 확인은 디바이스 에이전트가 메시지를 받아 처리할 수 있음을 뜻한다. 실행 결과는
제어 동작이 성공했는지, 실패했는지, 유효 시간이 지났는지를 뜻한다. 이 둘을 분리하면
운영자는 전송 실패, 에이전트 거절, 실행 실패를 서로 다른 문제로 다룰 수 있다.

방향과 의미도 분리해야 한다. `command`는 상대에게 부작용을 요구하는 메시지고,
`result`는 어떤 처리의 결론을 알리는 메시지다. 송신자와 수신자가 바뀐다고 메시지의
목적이 바뀌지는 않는다. 이 구분이 있어야 `...Result`라는 동작을 계속 추가하는 대신,
하나의 결과 메시지로 명령 생명주기를 표현할 수 있다.

## 결정 3: 상관관계와 연결 생명주기는 SDK가 맡는다

메시지 브로커는 HTTP처럼 요청과 응답을 자동으로 짝지어 주지 않는다. 명령을 보낸 뒤
늦게 도착한 수신 확인과 실행 결과를 원래 명령에 연결하려면 별도 설계가 필요하다.

SDK는 명령마다 식별자를 만들고 후속 메시지가 그 식별자를 참조하게 할 수 있다.
호출자는 전송 여부뿐 아니라 명령의 현재 상태도 읽을 수 있다.

```ts
// 설명을 위한 의사 코드
const command = await client.sendCommand({
  target: 'device-42',
  action: 'SLOW_DOWN',
  payload: { level: 'gentle' },
});

const accepted = await client.waitForAck(command.id, { timeoutMs: 5_000 });
const result = await client.waitForResult(command.id, { timeoutMs: 30_000 });
```

모든 업무 정책을 SDK가 결정할 필요는 없다. 타임아웃 뒤에 재시도할지, 중단할지,
운영자에게 알릴지는 장비와 업무의 위험도에 따라 애플리케이션이 정한다. SDK는 식별자,
대기, 중복 메시지 처리, 상태 전달처럼 반복되는 기계적 일을 맡는다.

이 글에서 다루는 SDK는 연결 생명주기도 책임진다. 연결이 끊기면 재시도하고, 다시
연결되면 필요한 구독을 복원하며, 실패를 애플리케이션이 처리할 수 있는 오류로 전달한다.
디바이스 에이전트마다 이 동작을 다시 구현할 이유는 없다.

## 결정 4: 전송은 어댑터로 감추고 계약으로 맞춘다

다국어 SDK에서 모든 언어의 연결 동작을 하나의 공통 구현으로 만들고 싶어질 수 있다.
하지만 언어별 런타임, 빌드, 배포 방식이 달라질수록 공통 코어의 유지 비용도 커진다.
더 작은 경계부터 시작하는 편이 안전하다.

```text
SDK 공통 계층: 메시지 조립, 상관관계, 라우팅 규칙, 검증 규칙
언어별 어댑터: 연결, 구독, 재연결, 전송 라이브러리 호출
디바이스 에이전트: 업무 동작과 페이로드 검증
```

언어마다 자연스러운 API는 다르다. 어떤 언어는 예외를, 다른 언어는 `Result` 값을,
또 다른 언어는 콜백이나 비동기 흐름을 중심으로 API를 만든다. 표면 API를 기계적으로
똑같이 만들면 각 언어의 사용성이 나빠질 수 있다.

대신 여러 언어가 같은 의미의 메시지를 만들고 해석하는지 검증해야 한다. 고정 계약
예제(fixture)를 계약의 정본으로 두면 다음을 자동으로 확인할 수 있다.

- 메시지 종류별 정상 입력과 기대 출력
- 선택 필드의 생략 규칙
- 잘못된 종류, 누락된 식별자, 끊어진 상관관계 같은 실패 사례
- 각 언어에서 직렬화한 메시지가 계약과 일치하는지 확인하는 테스트

이 방식은 한 언어의 라이브러리를 다른 언어에 이식하지 않아도 구현 간 차이를 발견하게
한다. [MAVSDK](https://github.com/mavlink/MAVSDK)는 C++ 기반 통신 계층, 공통 Proto IDL,
언어별 클라이언트를 분리한 공개 사례다. 특정 구조를 그대로 따르기보다, 공통 계약과
언어 관용적 클라이언트를 분리할 수 있다는 점을 참고할 수 있다.

## 결정 5: 실제 장비보다 먼저 시뮬레이터를 제공한다

실제 장비가 연결돼야만 개발할 수 있으면 SDK를 도입해도 팀의 대기 시간은 줄지 않는다.
전송 어댑터를 인터페이스로 분리한 뒤 인메모리 전송 어댑터를 제공하면, 장비 없이도
다음 시나리오를 재현할 수 있다.

- 연결 실패와 재연결
- 수신 확인이 오지 않는 명령
- 중복 수신과 중복 실행 방지
- 실행 결과 지연 또는 유실
- 잘못된 계약 메시지 거부

시뮬레이터는 테스트용 가짜에 그치지 않는다. 새 팀이 실제 장비를 기다리지 않고 통합
방법을 배울 수 있는 개발 환경이 된다. [Woven by Toyota의 Arene](https://woven.toyota/en/technology/arene/)도
SDK와 함께 가상 환경 및 검증 흐름을 제공하는 방향을 제시한다.

## 도입 성공 기준: 코드 양이 아니라 팀의 자립성

SDK 도입 효과를 줄어든 코드 줄 수로만 판단하면 중요한 변화를 놓친다. 아래 질문으로
팀의 자립성을 점검하는 편이 낫다.

- 새 디바이스 에이전트를 플랫폼 팀의 설명 없이 연결할 수 있는가?
- 새 업무 동작을 추가해도 SDK 릴리스가 필요 없는가?
- 명령이 전송, 수신, 실행 중 어디에서 멈췄는지 알 수 있는가?
- 같은 계약을 여러 언어에서 자동으로 검증하는가?
- 실제 장비가 없어도 통합 테스트를 시작할 수 있는가?

## 마무리

모두에 “예”라고 답할 수 있다면 SDK는 단순한 연결 래퍼가 아니다. 디바이스 에이전트
개발자가 통신 세부사항 대신 본연의 업무에 집중하도록 만드는 플랫폼 경계다.

표준화는 모든 구현을 하나로 합치는 일이 아니다. 여러 팀이 함께 지켜야 할 계약을
고정하고, 장비마다 달라져야 할 업무 데이터를 열어 두는 일이다.
