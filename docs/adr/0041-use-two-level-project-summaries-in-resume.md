# 0041. Resume 프로젝트는 짧은 소개와 상세 설명을 나눠 제공한다

Date: 2026-07-20
Status: Accepted

## 배경

요약 Resume은 핵심 키워드를 빠르게 보여주지만, `CDC analytics`나
`Platform redesign` 같은 영문 성과 라벨만으로는 실제로 무엇을 했는지 알기
어려웠다. 회사 열의 `Backend Engineer · IoT Systems`도 좁은 열에서 자연스럽지
않게 줄바꿈됐다.

공개 Resume은 상세 이력서보다 짧아야 하지만, 각 성과의 문제와 수행 내용을
질문할 수 있을 만큼의 문맥은 남겨야 한다.

## 결정

- 회사 메타 정보는 역할과 도메인을 별도 줄로 나눈다. 모노리스는 `백엔드
  엔지니어`, `IoT 시스템`으로, 엑심베이는 `소프트웨어 엔지니어`, `PG 플랫폼`으로
  표시한다.
- 각 프로젝트는 Resume 콘텐츠 열의 2열에 제목과 한 줄 소개를 둔다.
- 프로젝트의 상세 설명은 3-4열을 사용해 한국어 문장으로 표시한다.
- 중앙 콘텐츠 폭이 좁아지는 1,024px 미만에서는 4열 구성을 2열로 전환해
  한국어 단어가 글자 단위로 끊기지 않게 한다.
- 상세 설명에는 어떤 흐름을 바꿨는지와 검증 가능한 결과를 함께 쓴다. 긴
  문제·결정·구현·검증 단계 전체는 공개 Resume에 다시 도입하지 않는다.

## 결과

- 첫 훑기에서는 프로젝트 제목과 한 줄 소개를 읽고, 필요할 때만 같은 행의
  상세 설명을 이어 읽을 수 있다.
- 영문 기술 키워드만 나열하는 것보다 실제 업무 범위와 성과를 이해하기 쉽다.
- 공개 Resume은 요약본의 밀도를 유지하면서 면접 후속 질문에 답할 수 있는
  최소 문맥을 제공한다.

## 검토한 대안

- 성과 키워드만 유지한다: 가장 빠르지만 실제 수행 내용을 이해하기 어렵다.
- 기존 상세 이력처럼 단계별 서술을 전부 표시한다: 증거는 많지만 첫 화면이
  다시 길어진다.
- 프로젝트별 짧은 소개와 상세 설명을 한 행에 나눈다: 읽기 단계는 늘지만
  필요한 문맥을 가장 작은 범위에서 제공한다.

## 검증

- ResumePage 컴포넌트 테스트로 한국어 프로젝트 제목, 간략 소개, 상세 설명을
  검증한다.
- Playwright로 데스크톱 4열과 모바일 2열 전환에서 프로젝트 설명이 보이는지
  검증한다.
